# Your First Vehicle

Imagine a moth drawn to a porch light. It doesn't plan a route or make decisions — it simply moves faster when light hits its sensors. That's the idea behind a Braitenberg vehicle: wire a sensor directly to a motor and behavior emerges from the connection itself.

In this tutorial, you'll build the simplest version of this: a robot with one light sensor wired to both wheels. Shine a flashlight at it and it drives toward you. Cover the sensor and it stops. No code, no logic, no if-statements — just a direct connection.

This is essentially Braitenberg's **Vehicle 1**, the first machine in his 1984 book *Vehicles: Experiments in Synthetic Psychology*. His method was to *build* minds rather than dissect them: start with a trivial mechanism, set it loose, and watch how quickly we reach for words like "wants," "likes," and "afraid" to describe it. As you build, hold on to the gap between what the vehicle *is* — a sensor and two wires — and what it *appears to be doing*. That gap is the real subject of this whole series, and it only widens from here. (For the full tour of vehicles, see [Braitenberg Vehicles](../guide/braitenberg-vehicles).)

**Time:** 5 minutes
**Prerequisites:** BraitenBot installed and open

## What you'll build

A photoresistor (or any analog light sensor) on pin A0, wired directly to both wheel motors. The brighter the light, the faster the robot drives — both wheels at equal speed, so it travels in a straight line.

## Step 1: Observe the default canvas

When you first open BraitenBot, the canvas already has two nodes: **Left Wheel** and **Right Wheel**, snapped to the robot wheel overlay. These represent the two motors that drive the robot's wheels. Every vehicle starts here.

## Step 2: Add a sensor

1. In the **Node Palette** on the left, find the **Sensors** section
2. Drag **Analog Sensor** onto the canvas, dropping it above the robot
3. The sensor appears as an orange-bordered node labeled "Analog Sensor 1"

On the physical robot, this sensor is typically a photoresistor — a component whose resistance changes with light intensity. It outputs a voltage that the Arduino reads as a number. BraitenBot scales this to a simple 0–100 range for you: 0 means no light, 100 means maximum brightness.

## Step 3: Configure the sensor pin

1. Click the sensor node to select it — the config panel opens on the right
2. In the **Arduino Port** field, type `A0` — this must match the analog pin your sensor is physically wired to on the Arduino
3. Rename the label to "Light Sensor"

## Step 4: Connect sensor to motors

1. Hover over the sensor's **output handle** (the filled circle at the bottom of the node)
2. Click and drag down toward the **Left Wheel** node
3. When the Left Wheel's input handle glows, release to create the connection
4. Repeat: drag from the sensor's output to the **Right Wheel**

You should now see two curved lines connecting the sensor to both motors. Each connection carries the sensor's signal to a motor — this is the "wire" in Braitenberg's thought experiment.

## Step 5: Set connection weights

Both connections default to a weight of 1.0, which means the sensor signal passes to the motors unmodified. A sensor reading of 75 (bright light) becomes a motor drive of 75 (fast forward). A reading of 10 (dim) becomes a slow crawl.

To verify:
1. Click on a connection line (or its weight badge)
2. The config panel shows the **Connection Weight** slider (with a separate **Numeric Weight** box)
3. Confirm it's set to **1.0**

If you set the weight to 0.5, the motor would receive half the sensor signal — the robot would move more slowly for the same light level. We'll use this in later tutorials to tune sensitivity.

## Step 6: Test with trace simulation

Before uploading to hardware, you can simulate the signal flow right in the app:

1. Click **Trace Signal Flow** in the toolbar
2. A slider appears near your sensor node — this simulates the light level the sensor would read
3. Drag the slider from 0 to 100 and watch both motor values change in real time
4. At 0 (darkness): both motors read 0 — the robot is stopped
5. At 50 (moderate light): both motors read 50 — moderate forward speed
6. At 100 (bright light): both motors read 100 — full speed

Since both motors always get the same value, the robot drives in a straight line. It can't steer yet — we'll fix that in the next tutorial.

## Step 7: Upload to the robot

If you have an Arduino connected:

1. Set the **Loop** period to `20` ms in the toolbar (this means the robot reads its sensor and updates its motors 50 times per second)
2. Select each wheel motor (**Left Wheel** / **Right Wheel**) and set its **Servo Pin** — for example, `9` for the left wheel and `10` for the right (otherwise uploading fails with a "has no pin configured" validation error)
3. Select your board in the **Device** dropdown
4. Click **Upload to robot** — BraitenBot validates your diagram, generates the Arduino sketch, and uploads it to the board

If you don't have a board connected yet, you can still click the **▾** chevron next to the button, choose **Generate code only**, and then click the button to see the generated sketch.

Now try it: place the robot on a table and shine a flashlight at the sensor. The robot should drive toward the light source. Move the light and the robot follows — not by steering, but simply by speeding up or slowing down based on brightness.

:::note[Thinking like Braitenberg]

Watch someone meet this robot for the first time. They'll say it "likes" the light, or "is looking for" the flashlight — the language of goals and desires. But you built it, so you know there is no goal anywhere inside it: only a wire that turns brightness into speed.

Braitenberg called this the **law of uphill analysis and downhill invention** — it is far easier to *create* a behavior than to *deduce* the mechanism behind one. An observer reasoning backward from what the robot does would invent something far more elaborate than the truth. We do exactly this every day when we explain other people, and perhaps even ourselves. Sitting with that possibility is the real point of the exercise; the robot is just the excuse.
:::

## What you've learned

With just one sensor and two wires, you've created a vehicle that responds to its environment. There's no program logic — the behavior is entirely a product of the physical wiring. This is the core insight of Braitenberg vehicles: complex-looking behavior can emerge from very simple connections.

### Questions to sit with

- If a friend insisted the robot "wants" the light, what is the simplest experiment that would change their mind? Would it change *yours*?
- The behavior isn't really in the robot or in the room, but in their interaction. Where, exactly, is "moving toward light" located — in the sensor, the wire, the motor, or the flashlight?
- You wrote no rules, yet the robot reliably does something that looks purposeful. How much of a *living* animal's behavior, do you think, might also run without rules?

But this vehicle has a limitation: it can only drive straight. To make it steer toward a light source, you need two sensors — one on each side — with crossed wiring. That's what we'll build next.

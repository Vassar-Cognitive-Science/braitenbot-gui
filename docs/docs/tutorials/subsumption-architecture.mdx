# Subsumption Architecture

So far we've built individual behaviors: drive toward light, avoid obstacles, reverse on bump. But a real robot needs all of these working together. What happens when the cruise behavior says "forward" and the avoidance behavior says "turn right" at the same time?

If you look back at the [obstacle avoidance tutorial](./obstacle-avoidance), you've actually already solved this problem once. The Constant node provides a forward drive, and the threshold + negative weight connections override it when an obstacle appears. That's two behaviors — cruise and avoid — with avoidance suppressing cruise through inhibition. You built a two-layer subsumption architecture without knowing it.

In this tutorial, we'll make that structure explicit, then extend it to three layers by adding the bump-and-reverse escape behavior from the [latch tutorial](./latches-with-delay). The key principle is simple: **higher-priority behaviors suppress lower-priority ones when they activate.**

This tutorial quietly abandons an assumption you might not know you were making: that somewhere in a behaving robot there is a *decider* — a central place where options are weighed and a choice is issued. There isn't one here. Cruise, avoid, and escape all run at once, blind to each other, and "what the robot does" is simply whatever their signals sum to. As you wire it up, look for the spot where you expect a controller to live — and notice the robot works fine without it.

**Time:** 30 minutes
**Prerequisites:** Completed [Obstacle Avoidance](./obstacle-avoidance) and [Latches with Delay](./latches-with-delay)

## Part 1: Two layers — Cruise + Avoid

Let's start by rebuilding the obstacle avoidance circuit and labeling the layers explicitly.

### The Cruise layer

The cruise layer is the robot's default behavior: drive forward at a constant speed. It's just a Constant node wired to both motors.

1. Add a **Constant** node, value **50**, label "Cruise"
2. Add two **Summation** nodes: "Left Mix" and "Right Mix"
3. Wire: Cruise → Left Mix (weight 1.0), Cruise → Right Mix (weight 1.0)
4. Wire: Left Mix → Left Wheel (weight 1.0), Right Mix → Right Wheel (weight 1.0)

The summation nodes are where behaviors combine. Right now only cruise feeds into them, so both motors run at 50.

### The Avoid layer

The avoid layer watches for obstacles and steers away. It uses the cross-wired threshold inhibition pattern from the earlier tutorial.

1. Add two **ToF Distance** sensors: "Left Distance" (XSHUT 8), "Right Distance" (XSHUT 12)
2. Add two **Threshold** nodes: "Left Detect" (threshold 60), "Right Detect" (threshold 60)
3. Wire each sensor → its threshold (weight 1.0)
4. Cross-wire with negative weights: Left Detect → **Right** Mix (weight **-1.0**), Right Detect → **Left** Mix (weight **-1.0**)

### How suppression works here

Walk through the signal values in trace mode:

**No obstacles** — both thresholds output 0. Each Mix node receives only the cruise signal: 50. Both motors at 50. The robot drives straight.

**Obstacle on the left** — Left Distance reads 80, Left Detect fires at 100. The -1.0 weight sends -100 to Right Mix. Right Mix computes: 50 (cruise) + (-100) (avoid) = **-50**. The right wheel reverses while the left stays at 50. The robot pivots right, away from the wall.

This *is* suppression: the avoid layer's -100 overwhelms the cruise layer's +50, taking effective control of the right motor. Cruise is still running — it's still contributing +50 — but the avoidance signal is strong enough to override it. The cruise layer is suppressed, not turned off.

**Obstacle clears** — the sensor drops below threshold, the -100 disappears, and cruise's +50 takes over again seamlessly. No transition logic needed.

This is the essence of subsumption. Each layer runs independently and continuously. When a higher-priority layer has something to say, its signal is strong enough to dominate the summation. When it goes quiet, the lower layer's signal is all that's left.

## Part 2: Adding Escape — the third layer

The two-layer system handles most situations, but sometimes the distance sensors miss an obstacle — a thin chair leg, or a surface angled so the laser beam scatters away instead of bouncing back — and the robot drives right into it. That's where the bumper comes in.

The escape behavior is: **when the bumper is pressed, reverse both motors for about 2 seconds, overriding everything else.** This is the highest-priority layer. It must suppress both cruise and avoid.

### Building the escape latch

We need the decay latch from the [latch tutorial](./latches-with-delay): a bumper press charges a self-sustaining loop that decays over a couple of seconds.

1. Add a **Digital Sensor**: "Bumper" (pin 2, INPUT_PULLUP)
2. Add a **Summation** node: "Escape Latch"
3. Add a **Delay** node: set to **20 ms** (one loop period)
4. Wire: Bumper → Escape Latch (weight 1.0)
5. Wire: Escape Latch → Delay (weight 1.0)
6. Wire: Delay → Escape Latch — set the feedback to a **linear transfer with weight 0.98**. Each 20 ms tick the fed-back signal is 98% of the last, so a bump charges the sum to ~100 and it decays away geometrically. (Don't use a clamping curve here — a non-linear transfer ignores the weight, which would turn the loop into an *indefinite* latch that never releases.)
7. Add a **Threshold** node: "Escape Active" (threshold 10)
8. Wire: Escape Latch → Escape Active

When the bumper is pressed, the Escape Latch charges to ~100 and decays by ×0.98 each tick. It stays above the threshold of 10 for about 2.3 seconds — `loopPeriod × ln(10 / 100) / ln(0.98) ≈ 114 ticks × 20 ms` — so Escape Active outputs a clean 100 for ~2.3 s and then snaps back to 0 the moment the decaying sum drops below 10. Want a different hold time? Turn the feedback weight: 0.95 gives ~0.9 s, 0.99 gives ~4.6 s.

### Wiring escape into the mix

Escape needs to reverse both motors. Wire Escape Active → Left Mix (weight **-1.0**) and Escape Active → Right Mix (weight **-1.0**).

Connection weights are capped at ±1, so the strongest pull a single connection can apply is -100 (a 0–100 signal times weight -1.0). When escape fires with no obstacle present, each Mix computes 50 (cruise) + (-100) (escape) = **-50** — both motors reverse together. Escape doesn't need a larger weight to take priority. Unlike avoidance, which inhibits only the *opposite* motor (producing a turn), escape inhibits *both* motors symmetrically (producing a straight reverse), and the latch holds it for about 2 seconds. Escape's priority comes from its topology and timing, not from an out-of-range weight.

If avoidance is also active during escape, the motor on the obstacle side sees +50 (cruise) + (-100) (avoid) + (-100) (escape) = -150, which clamps to -100. The two inhibitory layers reinforce each other rather than fight — the robot backs away from the wall.

:::note[Why not -1.5?]

It's tempting to give escape a weight like -1.5 to "overpower" both lower layers at once, but that isn't possible — weights clamp to the range **-1 to +1**. A single connection can scale a 0–100 signal to at most ±100. To suppress a lower layer more aggressively than that, gate it (for example with a Multiply node) or restructure the layering — don't reach for a weight outside ±1.
:::

## Testing the complete system

Activate trace mode and walk through each scenario. The signal arithmetic at the Mix nodes tells you exactly what the motors will do.

### Scenario 1: Open floor (cruise only)

All sensors at 0. No thresholds firing, latch inactive.

| Input to Left Mix | Value |
|---|---|
| Cruise | +50 |
| **Total** | **50** |

Both motors at 50. The robot drives forward.

### Scenario 2: Wall on the left (avoid suppresses cruise)

Left Distance at 80 → Left Detect fires at 100.

| Input to Right Mix | Value |
|---|---|
| Cruise | +50 |
| Left Detect (cross-wired) | -100 |
| **Total** | **-50** |

Left motor stays at 50, right motor at -50. The robot pivots right. Avoidance has effectively taken over the right motor.

### Scenario 3: Bumper hit (escape suppresses everything)

Pulse the bumper. Escape Active fires at 100 while the latch decays — about 2.3 seconds. No obstacles present.

| Input to Left Mix | Value |
|---|---|
| Cruise | +50 |
| Escape Active | -100 |
| **Total** | **-50** |

Both motors at -50. The robot reverses straight back (both wheels together). The Escape Latch's sum decays by ×0.98 each tick; once it slips below the threshold of 10 — about 2.3 seconds after the bump — Escape Active snaps to 0 and the motors return to 50. (A longer or firmer press charges the latch higher, so it holds a little longer.)

### Scenario 4: Bumper hit *while* avoiding an obstacle

Bumper pressed and Left Distance at 80 simultaneously.

| Input to Right Mix | Value |
|---|---|
| Cruise | +50 |
| Left Detect (avoid) | -100 |
| Escape Active | -100 |
| **Total** | **-150 → clamped to -100** |

| Input to Left Mix | Value |
|---|---|
| Cruise | +50 |
| Escape Active | -100 |
| **Total** | **-50** |

Right motor at -100, left motor at -50 — both wheels reverse, the right one harder because avoidance reinforces escape on that side. The robot backs away from the wall, curving as it goes. The two inhibitory layers stack rather than fight: escape sets the overall reverse, and avoidance biases the turn.

### Scenario 5: Escape ends, obstacle still present

Once the Escape Latch decays below the threshold — about 2.3 seconds after the bump — Escape Active drops to 0. Left Distance still reads 80.

| Input to Right Mix | Value |
|---|---|
| Cruise | +50 |
| Left Detect (avoid) | -100 |
| Escape Active | 0 |
| **Total** | **-50** |

The robot immediately begins steering right — avoidance picks up seamlessly where it left off. There's no gap, no transition logic. The layers just produce their signals independently, and the summation sorts out which behavior dominates.

## Why this works

The key insight is that suppression doesn't require any explicit "if-then" logic or mode switching. Every layer runs all the time, producing its output signal into the shared summation. Priority is encoded in the *weights* and the *wiring topology*:

- Cruise contributes +50 (moderate forward)
- Avoid contributes -100 to one motor — strong enough to override cruise and pivot the robot
- Escape contributes -100 to *both* motors — reversing the robot — and the decay latch holds it for about 2 seconds (set by the feedback weight)

An inhibitory -100 connection always overrides the +50 cruise, so any active layer takes effective control. Escape sits at the top not because of a bigger weight (weights cap at ±1) but because it inhibits both wheels at once and sustains itself through the latch. The summation acts as the arbitration mechanism: whichever layers are active sum together, and when the strong inhibitory signals disappear, cruise is automatically revealed.

This is subsumption's central virtue: **behaviors are independent modules that don't know about each other.** The avoid layer doesn't check whether escape is active. Escape doesn't turn off avoidance. They just produce signals, and the math sorts it out. You can add or remove layers without modifying the existing ones.

:::note[Intelligence without a boss]

No part of this robot is "in charge." No layer knows the others exist; none can switch another off by command. Priority isn't a decision anyone makes — it falls out of topology and arithmetic, the way -100 simply outweighs +50 at a summation. The competent-looking whole has no center.

This is the radical idea Rodney Brooks built behavior-based robotics on in the 1980s: **intelligence without representation**. Skip the central model of the world and the planner that reasons over it; layer many simple, tightly-coupled sensor-to-motor behaviors and let conflicts resolve at the wheels, not in a mind. It works unnervingly well — and it raises a question that won't quite go away: if a robot can be this competent with nobody home, how sure are you that *you* have a boss inside, rather than a very good committee?
:::

## The physical result

Put the robot in a room with furniture and watch:

1. It drives forward (cruise)
2. It approaches a table leg — the distance sensors detect it and the robot veers away (avoid overrides cruise)
3. It clips a corner the sensors missed — bumper clicks — the robot reverses for about 2 seconds (escape overrides everything)
4. After reversing, it resumes driving forward. If the table leg is still nearby, avoidance kicks in again immediately

The robot navigates surprisingly well for something with no planning, no map, and no centralized control. Each behavior is a simple sensor-to-motor circuit. The competence comes from the layering.

## Extending the architecture

Once you have this pattern, adding new behaviors is straightforward:

- **Wander layer** (priority between cruise and avoid): add an Oscillator node with a symmetric dead-zone transfer curve feeding into the Mix nodes with weight ±0.3. The robot will gently weave left and right while cruising, covering more area. Avoidance overrides it because -100 > ±30.

- **Light seek layer** (priority between avoid and escape): add the two light-sensor boards with cross-wired connections to the Mix nodes, weight ±0.8. The robot drives toward light when no obstacles are in the way.

For each new layer, choose a weight magnitude that's stronger than the layers it should suppress and weaker than the layers that should suppress it.

## Packaging behaviors as compounds

The diagram has grown complex — 13 nodes. You can clean it up by grouping each behavior into a compound:

- **Escape** compound: contains the bumper, latch, delay, and threshold. Exposes one output: the -100 suppression signal sent to both motors.
- **Avoid** compound: contains the two distance sensors, two thresholds, and the cross-wired connections. Exposes two outputs: left and right inhibition signals.

See [Compound Reuse](./compound-reuse) for how to do this.

### Questions to sit with

- Nothing in this robot chooses between behaviors, yet it reliably does the "right" one. Does arbitration require an arbiter — or just the right place for signals to collide?
- Each layer is ignorant of the others. Would the robot be *better* with a central controller that knew about all of them, or would you just be adding a single point of failure and calling it a mind?
- Watching the finished robot dodge, reverse, and resume, you'll instinctively narrate one continuous agent. But it's three deaf modules summing into a wheel. Which is the illusion — the unified agent, or the separateness of the parts?

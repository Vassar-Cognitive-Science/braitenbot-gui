# Latches with Delay

Put a bumper switch on the front of the obstacle-avoidance robot from the earlier tutorial and you'll discover a problem. The robot detects the wall with its distance sensors and turns away — but sometimes it doesn't turn far enough and clips the corner. The bumper clicks for a fraction of a second, then releases. The robot doesn't react because by the time the next loop iteration runs, the bumper is already open again. The event is gone.

What you need is **memory**: a way for the robot to notice a brief event (the bumper click) and hold onto it for long enough to respond (reverse for two seconds). This tutorial introduces the latch — a feedback loop built from a Summation and a Delay node that captures transient events and sustains them.

There's a deeper puzzle hiding here. Every circuit you've built so far is **purely reactive**: its output depends only on what the sensors read *right now*. A vehicle like that lives in an eternal present — it has no "a moment ago." And its sensors make this worse, not better: they report the present *continuously*, so a 50 ms bump is, on the very next reading, simply gone. Nothing in the circuit ever saw a "then."

So where could a memory possibly live? Not in any single part — none of them store anything. The answer turns out to be **topology**: route a signal back into itself through a delay, and the loop keeps that signal alive after its cause has vanished. Memory becomes a *process* rather than a place — activity sustaining itself. You're about to build "the past" out of components that can only ever sense "now."

**Time:** 20 minutes
**Prerequisites:** Completed [Obstacle Avoidance](./obstacle-avoidance) and familiar with [signal flow](../concepts/signal-flow)

## The scenario: bump and reverse

Here's the concrete behavior we're building:

1. The robot drives forward at cruising speed
2. Its front bumper hits a wall (momentary contact — maybe 50ms)
3. The robot remembers the bump and reverses for 2 seconds
4. After 2 seconds, the robot stops reversing and drives forward again

The bumper is a momentary switch — it's only pressed for a few loop iterations. Without memory, the robot would reverse for those few iterations (imperceptibly) and then immediately drive forward again, right back into the wall.

## How a latch works

A latch is a feedback loop: a Summation node sends its output through a Delay node, which feeds the value back into the Summation one iteration later.

Walk through the loop iterations:

1. **Before the bump**: Summation receives 0 from the bumper and 0 from the delay feedback. Output is 0. Motor is off.
2. **Bumper pressed**: Bumper sends 100. Summation outputs 100. The delay captures 100.
3. **Bumper released**: Bumper sends 0. But the delay feeds back the *previous* output (100). Summation receives 0 + 100 = 100. The latch holds.
4. **Every subsequent iteration**: The delay keeps feeding back 100, and the summation keeps outputting 100. The latch remembers the bump indefinitely.

The robot's motor stays active even though the bumper was only pressed for a moment. The feedback loop sustains the signal.

:::note[Memory without a memory]

Look closely at what just happened: the latch "remembers" the bump, yet nowhere in the circuit is there a part that *holds* the value 100. The summation only ever adds its inputs in the current instant; the delay only ever echoes what it heard one tick ago. The memory isn't stored anywhere — it lives in the **circulation**, a signal handed from delay to summation and back, around and around.

This is a strange idea, and it isn't just a trick of the editor. Many models of biological short-term memory work the same way: a memory is held not by a dedicated "storage" cell but by a loop of neurons that keep re-exciting each other. Break the loop for a single tick and the memory is gone — instantly and completely, with no trace left behind. A thought, on this view, is something a circuit *does*, not something it *has*.
:::

## Step 1: Build the latch

1. Drag a **Summation** node onto the canvas, label it "Latch"
2. Drag a **Delay** node onto the canvas — set its delay to **20 ms** (one loop period, the minimum for a simple latch)
3. Connect Summation → Delay (weight 1.0)
4. Connect Delay → Summation (weight 1.0)

## Step 2: Add the bumper trigger

1. Drag a **Digital Sensor**, set port to `2`, label "Bumper"
2. Connect Bumper → Summation (weight 1.0)

On the physical robot, wire a pushbutton between pin 2 and ground, and enable INPUT_PULLUP on the sensor node. The button reads as 100 in BraitenBot when pressed (pullup mode inverts the signal so that pressing the button produces a HIGH reading).

## Step 3: Connect to a motor

Connect Summation → Right Wheel (weight 0.5) to see the latch in action.

## Step 4: Test with trace mode

1. Activate **Trace Signal Flow**
2. Click the **pulse button** (▶) next to the bumper — this sends a brief 200ms HIGH, simulating a momentary button press
3. Watch the latch activate: the summation jumps to 100 and the motor to 50
4. After the pulse ends, the bumper returns to 0 — but the latch holds. The motor stays at 50.

The pulse button is specifically designed for testing circuits like this. It simulates the brief, transient events that you'd get from real bumper switches or momentary buttons.

## Fixing the runaway problem

You may notice the summation value keeps climbing: 100, 200, 300... Each iteration adds the feedback (100) to whatever the summation already is. This runaway doesn't affect the motor (it still drives at full speed), but it means the latch is numerically unstable.

Fix it with a **clamping transfer curve** on the feedback connection:

1. Click the Delay → Summation connection
2. Switch to Non-linear transfer mode
3. Set a step curve: points at (-100, 0), (0, 0), (1, 100), (100, 100)

Now the feedback is always exactly 0 or 100 — any positive value clamps to 100, any negative to 0. The summation stabilizes at 100 when latched and 0 when cleared.

## Making the latch time out

An indefinite latch isn't useful for bump-and-reverse — you want the robot to reverse for a specific time and then drive forward again. Two approaches:

### Approach 1: Decay feedback

Change the feedback weight from 1.0 to **0.95**. Each iteration, the feedback is 95% of the previous value:

100 → 95 → 90.25 → 85.7 → 81.5 → ... → eventually reaches 0

Add a **Threshold** node (threshold = 10) after the summation to convert the decaying signal into a clean on/off. With a 20ms loop period, the signal drops below 10 after about 45 iterations — roughly 900ms. To hold for longer, use a smaller decay: a weight of 0.97 gives about 1.5 seconds, 0.98 gives about 2.3 seconds. Adjust the weight and threshold to control the duration.

### Approach 2: Longer delay

Set the delay time to **2000 ms** and use the clamping feedback from above. The delay buffer holds 100 iterations of history (2000ms / 20ms). When the bumper triggers, the latch holds for exactly 2 seconds (the time it takes the initial trigger to propagate through the buffer and disappear), then the feedback goes to zero and the latch releases.

This is more precise than decay feedback and easier to reason about: delay time = hold time.

:::tip[Two ways to forget]

These two timeouts aren't just implementation details — they're two different theories of forgetting. **Decay** is a *forgetting curve*: the trace fades smoothly, vivid right after the event and gradually slipping away, the way a striking moment dims over seconds. **The delay line** is an *echo*: the event is held perfectly intact for a fixed interval and then drops out all at once, like a tape loop reaching its end. Nervous systems appear to use both — graceful decay for some things, precise timing for others. Which one you reach for depends on whether the robot needs to know *how long ago* something happened or *exactly when* it should stop mattering.
:::

## Building the complete bump-and-reverse

Now combine the latch with the obstacle avoidance circuit:

1. **Cruise circuit**: Constant (50) → Left Summation and Right Summation → Motors (same as obstacle avoidance)
2. **Latch circuit**: Bumper → Summation → Delay → feedback, with a 2000ms delay and clamping feedback
3. **Inhibition**: Wire the latch's Summation output through a Threshold (threshold = 10) → both motor summations with weight **-1.0**

The physical behavior plays out like this:

- **Normal driving**: The latch is off. The constant drives both motors at 50. The robot cruises forward.
- **Bump happens**: The bumper clicks for 50ms. The latch captures it and holds for 2 seconds. The threshold fires, sending -100 to both motor summations. Motors get 50 + (-100) = -50 — the robot reverses.
- **2 seconds later**: The latch times out, the threshold turns off, the inhibition stops. Motors return to 50 and the robot drives forward again.

From the outside, the robot bonks into a wall, backs up for two seconds, and then resumes driving. The brief bumper contact got amplified into a sustained behavioral response — exactly what memory is for.

## Resettable latch

Sometimes you want to reset a latch manually — for example, a "panic button" that cancels an automated behavior. Add a second Digital Sensor ("Reset", port `3`) and connect it to the latch summation with weight **-1.0**:

When Reset fires it contributes 100 × -1.0 = -100, exactly canceling the +100 of clamped feedback. The summation drops to 0, the delay captures 0, and on the next iteration the feedback is 0 — the latch clears immediately, regardless of how much time was left.

:::note[Weights are limited to ±1]

Connection weights are clamped to the range **-1 to +1**. A single connection can therefore scale a 0–100 signal to at most ±100. To "overpower" a signal more strongly than that, use a [non-linear transfer curve](./transfer-curves), a Threshold, or stack multiple connections — not a weight outside ±1.
:::

## What you've learned

The latch pattern — summation + delay feedback — converts a transient event into a sustained state, giving the robot a form of short-term memory. This is what separates reactive behaviors (turn away from obstacles) from deliberate behaviors (reverse for two seconds after a bump).

### Questions to sit with

- The vehicle now does something its sensors cannot: it responds to an event that is no longer happening. Is that enough to say it *remembers*? What more, if anything, would you want before using that word?
- A reactive vehicle is fully explained by its present input. Add a loop and the *same* input can produce different behavior depending on what came before — the vehicle has **state**. Is that the first faint glimmer of an inner life, or just bookkeeping?
- You built "two seconds ago" out of parts that only know "now." Does that make you more impressed by what a brain full of such loops might do — or less?

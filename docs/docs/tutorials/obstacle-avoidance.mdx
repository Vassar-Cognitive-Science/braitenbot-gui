# Obstacle Avoidance

The light follower is elegant, but it has a problem: it drives straight into things. Put a box between the robot and the light source and the robot plows right into it — it has no concept of "obstacle."

In this tutorial, you'll add obstacle detection using the robot's two ToF distance sensors and a new building block: the **Threshold** node. The result is a robot that drives forward exploring its environment and turns away when it gets too close to a wall or object — the most common behavior in hobby robotics.

With this vehicle a new ingredient enters: a *negative* connection. Until now every wire urged a wheel to spin faster; here, detecting an obstacle tells a wheel to back off. That small change — a minus sign — is the seed of something that looks a lot like an aversion, a preference, a "don't." Watch how quickly "avoids walls" starts to feel like "doesn't want to crash," when all you've really added is subtraction.

**Time:** 15 minutes
**Prerequisites:** Completed [Light Follower](./light-follower)

## The physical setup

Mount the robot's two **ToF distance sensors** (VL53L4CD) on the front, angled outward about 30° from center — one pointing front-left, one front-right. Each one measures the actual distance to whatever is in front of it with a tiny time-of-flight laser and reports it over the shared I2C bus. The **ToF Distance** node scales that reading to 0–100, with a *closer* object reading *higher*, up to a maximum distance you choose.

Because both sensors sit on the same I2C bus and ship with the same default address, each needs its own **XSHUT** pin so BraitenBot can wake them one at a time at startup and give each a unique address (see [Supported Hardware](../reference/supported-hardware)). Note that, unlike a raw analog proximity sensor, the ToF reading is clean and genuinely proportional to distance — so why introduce a Threshold at all? Because for *avoidance* you rarely want a proportional response; you want a crisp decision: is something close enough to worry about? The Threshold node turns the continuous distance into exactly that yes/no.

## How it works

The design combines two ideas from previous tutorials:

1. **Constant forward drive** — a Constant node feeds both motors equally, so the robot always wants to drive forward at 50% speed.
2. **Cross-wired inhibition** — when a threshold detects an obstacle, it sends a *negative* signal to the opposite motor, subtracting from the constant drive. This creates differential steering away from the obstacle.

The Threshold node is the new piece. It watches its input and outputs either 0 (nothing detected) or 100 (obstacle detected). The negative weight on the cross-wire turns that 100 into -100 at the motor summation, which subtracts from the constant 50 and produces -50 — the motor reverses.

:::note[Where does "wanting to explore" come from?]

Watch this robot for a while and you'll narrate it generously: it *explores* the room, *wary* of walls, *drawn* to open space. But there is no exploration drive anywhere in the circuit — only a constant pushing it forward and a pair of negative wires pushing back when something's close. "Exploration" is simply what forward-plus-avoid *looks like* from the outside.

This is inhibition doing the work that, in our own heads, we file under motives and values. A negative number isn't a fear or a dislike — but wire enough of them together and you get a vehicle whose behavior is most easily *described* in the language of fears and dislikes. Braitenberg's wager is that this isn't a cheat or a coincidence: it's a hint about where our own preferences might bottom out.
:::

## Step 1: Set up the baseline drive

1. Drag a **Constant** node onto the canvas, set its value to **50**
2. Add two **Summation** nodes — one for each motor
3. Wire Constant → both Summations (weight 1.0), then each Summation → its Motor (weight 1.0)

Test in trace mode: both motors should show 50. The robot drives straight forward.

## Step 2: Add obstacle sensors

1. Drag two **ToF Distance** nodes onto the canvas, labeled "Left Obstacle" and "Right Obstacle"
2. Give each its own **XSHUT pin** — for example `7` and `8`. Both sensors share the I2C bus (SDA/SCL); the XSHUT lines let BraitenBot bring them up one at a time at startup and assign each a unique address. A closer object reads higher, ramping to 0 at the sensor's max distance.

## Step 3: Add threshold nodes

1. Drag two **Threshold** nodes onto the canvas
2. Wire each sensor to its threshold (weight 1.0)
3. Set both thresholds to **60**

The threshold value determines how close an obstacle must be before the robot reacts. Because the ToF reports distance, this is easy to reason about: with the sensor's max distance left at 500 mm, a closer object reads higher, so a threshold of 60 fires at roughly 200 mm (about 20 cm). To calibrate on the real robot, hold an object at the distance you want it to start turning, read the value in trace mode, and set the threshold just below it.

## Step 4: Cross-wire with negative weights

1. Connect Left Detect → **Right** Summation (weight **-1.0**)
2. Connect Right Detect → **Left** Summation (weight **-1.0**)

The cross-wiring ensures the robot turns *away* from the obstacle — same principle as the light follower, but inverted (negative weight means inhibition instead of excitation).

## Step 5: Test the behavior

Activate trace mode and walk through these scenarios:

**No obstacles** — both sensors at 0, thresholds output 0, both motors at 50. The robot cruises forward.

**Wall on the left** — left sensor at 80 (above threshold), Left Detect fires at 100. Right motor gets 50 + (-100) = -50. Left motor stays at 50. The right wheel reverses while the left drives forward — the robot spins right, away from the wall.

**Wall on the right** — mirror image. The robot spins left.

**Wall dead ahead** — both sensors fire, both motors get 50 + (-100) = -50. The robot backs straight up. In practice, it will back up until the sensors clear the obstacle, then resume driving forward.

## Calibrating for your hardware

### Threshold value

The right threshold depends on your specific sensors and how they're mounted:

- **Too low** (e.g., 20): the robot reacts to distant objects and never gets close to anything — overly cautious, may get stuck oscillating
- **Too high** (e.g., 90): the robot doesn't react until it's almost touching the obstacle — may clip corners
- **About right** (e.g., 50–70): the robot turns about 15–20 cm from obstacles

To find your value: put the robot near a wall, activate trace mode, and watch the sensor readings. Note the value at the distance where you want the robot to start turning.

### Inhibition weight

The weight on the inhibitory connections controls how sharply the robot turns:

- **-0.5**: the opposite motor slows to 0 but doesn't reverse — gentle, wide turns. Good for open spaces.
- **-1.0**: the opposite motor reverses to -50 — sharp pivot turns. Good for tight spaces like corridors.

Weights are limited to the range -1 to +1, so **-1.0 is the sharpest single-connection inhibition available**. If you need an even harder turn, lower the Cruise constant (so there's less forward drive to overcome) rather than reaching for a weight below -1.

## What you've learned

This tutorial introduced two important patterns:

- **Threshold nodes** convert a continuous sensor reading into a binary decision — turning a smooth distance measurement into a crisp "too close?" that drives a clean avoidance response
- **Negative weights** create inhibition — one part of the circuit suppressing another. This is the basis for more complex behaviors like priority arbitration in [subsumption architectures](./subsumption-architecture).

### Questions to sit with

- The robot "explores" with no urge to explore. How many of your own behaviors might be side effects of simpler pushes and pulls, rather than the goals you'd name for them?
- A positive wire is "approach" and a negative wire is "avoid." Is there anything more to a value than which way it points the motors — or is that genuinely all a value is, at the bottom?
- The robot never decides to turn; the numbers just sum to a turn. Did *anything* in there make a choice? If not, what would have to change before you'd say it did?
